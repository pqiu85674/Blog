Git Branch

假設 A 電腦跟 B 電腦都共用一個 Repositories
在 A 及 B 電腦上先clone
以下方便區分 A 及 B 電腦
所以將 A 電腦路徑設為./gitA/

B 電腦路徑設為./gitB/

先把 A 電腦 clone 下來的檔案內容進行更改
更改完後執行 git add . git commit -m 'A Message' git push
push 完之後看一下 log git log --oneline 可以看到 commit 的紀錄
這邊是故意先 push 讓其他使用者的 git log不同

剛建立好 branch 直接 push 會失敗，因為電腦不知道你要 push 哪個 branch，所以要設定git push -u origin HEAD同等於git push -u origin new-branch-name告訴電腦要 push 的 branch，HEAD 會抓到電腦的 branch 最後一次 commit，也可以直接打 branch 的名稱

到 Github 看你的 branch 就會看到兩個 branch
可以切換到 new-branch-name 點開 commit 可以看到 B 電腦所 push 的內容，也可以查看剛剛 B 電腦修改的檔案
看好之後切回 new-branch-name 可以看到上面綠色的Compare & pull request點擊下去
點擊 Create pull request
可以看到上面有base可以選擇main(原本的 branch) compate 選擇 new-branch-name(新的 branch)
會出現以下錯誤

Can’t automatically merge. Don’t worry, you can still create the pull request.

這邊沒辦法直接把新的 branch 合併到原本的 branch

這邊回到 B 電腦

會顯示檔案衝突要手動解決檔案的內容，這邊是使用 VS Code 直接開啟檔案就可以對檔案進行更改
修改完檔案後

這裡git push失敗的原因是因為剛剛我們在 B 電腦上把 A 電腦的修改給 pull 到 B 電腦上進行合併，所以目前的 B 電腦 git log 是 main 的 commit 合併 new-branch-name 跟 Github 裡面 new-branch-name 的 commit 不一樣，所以不能git push

這裡要用 git push -f 強制 push，就可以把我們合併的 commit 給 push 到 Github 的 new-branch-name

注意

這裡使用 git push -f 是將合併好的 commit 給 push 到 B 電腦所創造出來的 new-branch-name分支上
但是git push -f 千萬不要在主要分支做使用，只能在自己建立的 branch 分支上做使用，因為此指令會更改歷史紀錄，一旦在主要分支上使用會更改到別人的紀錄，會非常混亂

再次回到 Github 並切換到 new-branch-name 點開 commit 可以看到已經成功更新了
看好之後切回 new-branch-name 可以看到上面綠色的Compare & pull request點擊下去
點擊 Create pull request
可以看到上面有base可以選擇main(原本的 branch) compate 選擇 new-branch-name(新的 branch)
看到以下訊息
Able to merge. These branches can be automatically merged.
代表已經可以把新建立的分支放到主要分支上了
按下 Create pull request
按下 Merge pull request
按下 Confirm merge
這樣就成功了
可以查看 main 裡面的 commit

以 Node.js 實作一個 Server

  1. 建立 server.js 檔

    mkdir ‘資料夾名稱’
    cd ./資料夾名稱
    touch server.js
    code .

  2. 啟動伺服器

  3. 連線
    伺服器正常運作後可以開啟瀏覽器，輸入網址http://127.0.0.1:8000 or localhost:8000 就可以看到 Hello

  4. 使用 nodemon 套件
    每次修改儲存後都需要重起 node，可以透過 npm 安裝 nodemon 這個套件，就能在編輯檔案後自動重新載入伺服器。

  5. port 已經被使用

    若是出現 Error: listen EADDRINUSE: address already in use :::8000 錯誤
    代表 8000 port 正在使用

  6. 連線請求

    程式碼說明

    http： 引入 Node.js 的 HTTP，用於創建伺服器
    headers： 定義 CORS，允許跨域請求

    “Access-Control-Allow-Headers”
    作用：指定哪些 HTTP 的 Headers 設定可以在跨網域請求中發送到我的伺服器
    内容：”Content-Type, Authorization, Content-Length, X-Requested-With”
    這意味著客戶端可以在後續的實際請求中使用 Content-Type, Authorization, Content-Length, X-Requested-With

    “Access-Control-Allow-Origin”
    作用：指定允許哪些網域來源可以造訪我的伺服器
    内容：*
    表示所有的網域都可造訪我的伺服器

    “Access-Control-Allow-Methods”
    作用：指定允許客戶端使用的 HTTP 方法
    内容：”PATCH, POST, GET,OPTIONS,DELETE”
    表示允許使用 PATCH、POST、GET、OPTIONS 和 DELETE 方法。

    “Content-Type”
    作用：指定回傳內容型別
    内容：”application/json”
    客戶端回傳的數據型別為 JSON 格式

    requestListener ： HTTP 的 Request 以及 Response，狀態碼設為 200，內容為 Hello
    http.createServer(requestListener) ： 創建伺服器
    port ： 從環境變量讀取 port，如果沒有設置則默認使用 8000 port
    server.listen ： 啟動伺服器並且監聽 port 啟動成功輸出 Server is running on port 8000
    shutdown ： 定義一個函數，當接收到退出信號時關閉伺服器並退出
    process.on("SIGTERM", shutdown)和process.on("SIGINT", shutdown) ： 監聽終止信號，並調用 shutdown 關閉伺服器

    在nodemon server.js啟動伺服器想要關閉時按下ctrl+c就是觸發process.on("SIGINT", shutdown)
    而我也可以在終端機輸入kill -9 00000強置關閉你正在使用的 port 會觸發process.on("SIGTERM", shutdown)
    而使用這兩種SIGTERM和SIGINT可以優雅且關閉伺服器且沒有任何數據丟失或損壞非常重要

基本知識

Global

網頁環境中的全域物件是 window，而 Node.js 中是 global

  • 瀏覽器

  • Node.js 中的 global

    兩個檔案（app.js 和 app2.js），裡面都分別撰寫了 var a = 1，var b = 1，這時網頁的 window.a 和 window.b 會得到 1 跟 2 的結果，Node.js 的 global.b 和 global.b 則是 undefined。

    Node.js 中的每個模組都會是獨立的，不會互相干擾也不會變成全域變數（除非直接操作 global 物件）
